# Implementation Rules

## Назначение

Этот файл фиксирует правила, которые сложились в ходе разработки `appslides` и которые нужно сохранять при переносе проекта под другую тематику.

Это не README для пользователя. Это рабочая инструкция для разработчика или следующей нейросети.

## 1. Продуктовые инварианты

- Основной клиентский интерфейс — единый chat UX, а не набор вкладок.
- Пользовательский сценарий должен жить в одной ленте сообщений.
- Сообщения, кнопки, статусы и файлы должны быть частью одной беседы.
- История чата должна сохраняться локально и восстанавливаться после перезапуска приложения.
- Результаты генерации или обработки должны открываться как реальные локальные файлы.

## 2. Что нельзя ломать без явного решения

- `client_id` как ключ установки/клиента;
- локальное хранение transcript;
- локальный индекс файлов;
- billing state на backend;
- отдельный Telegram admin bot;
- GitHub -> deploy -> restart flow;
- mounted server directories `data/templates/temp/logs`.

## 3. Клиентские правила

- Любая новая тематика должна встраиваться в существующий chat shell, а не заменять его на dashboard без явного запроса.
- Временные progress-сообщения должны удаляться, когда приходит финальный результат.
- В чате не должно быть декоративных, но неработающих элементов.
- Если меняется структура сообщений, нужно проверить восстановление transcript после перезапуска.
- Если меняется file flow, нужно проверить скачивание, открытие и удаление файлов на Android.

## 4. Backend-правила

- Длинные операции должны жить через jobs или другой фоновый механизм, а не через долгий request-response.
- Платежи, лимиты, подписки и entitlement state должны оставаться серверными.
- Продуктовые prompts и генерация могут меняться, но billing/storage/admin-слой лучше не переписывать без необходимости.
- Если меняется продуктовый результат, нужно явно проверить artifact flow: создание, хранение метаданных, download route, открытие файла на клиенте.

## 5. Billing-правила

- Текущий платежный контур — `YooKassa`.
- Платежи идут через backend, а не напрямую через Flutter.
- Успешная оплата должна активировать entitlement без ручного вмешательства.
- Автосписания и продления должны логироваться и отправляться админам.
- Все важные billing-события должны иметь admin-уведомления.

## 6. Admin-правила

- Админка живет не внутри Flutter-клиента, а в отдельном Telegram admin bot.
- Проверка админов идет через `.env` и базу.
- В статистике и уведомлениях надо показывать полезные данные, а не лишний технический шум.
- `client_id` можно сокращать в отображении, но нельзя терять полный идентификатор в бизнес-логике.

## 7. Операционные правила

После каждого большого или важного изменения:

1. Прогнать релевантные локальные проверки.
2. Зафиксировать изменения в Git.
3. Запушить изменения в GitHub.
4. Если менялся backend, перевыкатить сервер и перезапустить контейнеры.
5. Обновить документацию и рабочий план.

Канонический файл по операциям:

- [OPERATIONS.md](OPERATIONS.md)

## 8. Документационные правила

- Не писать документы только под одну тему вроде Таро.
- Использовать формулировки вида `под твою тематику`, `новый продукт`, `новая ниша`.
- Явно разделять:
  - что является платформой;
  - что является предметной логикой;
  - что нужно менять;
  - что нужно сохранять.
- Для следующей нейросети всегда оставлять handoff-файлы, а не только технические README модулей.

## 9. Что проверять после тематической миграции

- стартовый сценарий;
- основной чат-флоу;
- генерацию/обработку результата;
- платежи;
- transcript restore;
- file open flow;
- admin-уведомления;
- deploy и healthcheck.

## 10. Базовый принцип

`appslides` нужно воспринимать как платформенный каркас.

Менять нужно прежде всего:

- контент;
- prompts;
- продуктовые статусы;
- брендинг;
- тип результата.

Сохранять нужно прежде всего:

- UX-каркас;
- storage-каркас;
- billing-каркас;
- admin-каркас;
- deploy-каркас.
